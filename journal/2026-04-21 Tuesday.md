- I started reading about about file permissions and understood that there are three types of permissions - read, write and execute
- Each permission has 10 characters. Character 1 signifies if the file type is a regular file (-) or a directory/folder (d), character 2-4 signifies owners permission, characters 5-7 signifies group permissions while characters 8-10 signifies every other person permission. For instance, `-rw-r--r--` tells that it is a regular file, owner has read and write permission while group and evey other person only have read permission. Another instance is `drwxr-xr-x` signifies that it is a folder, owner has read,write and execute permissions while group and everyone else has read and execute permissions only.
- I also learnt that every permissions has a numeric value. Read - 4, Write - 2 and Execute - 1. Hence, numbers like 7 would grant full permissions, 5 would grant read and execute permissions while 4 would grant read permission and 0 would grand no permission at all.
- Hence since we know how the numbers work, we can grant permission to files and folders using `chmod`. For instance, to grant full access to a folder named testFile, we can use `chmod 777 testFile`,  to grant full access to only the owner and read and execute access to group and everyone, we can use `chmod 755 testFile`. Therefore, for SSH keys and password files, the correct permission would be `chmod 600 secret.txt` which gives read and write permission to only the owner and nothing else to group and every other person.
- I learnt that SUID which stands for Security User ID is a special permission that allows a file to run as its owner regardless of who runs it. For instance, if root created a file and has SUID set, then anyone who run the file would run it as root user. This is a major attack surface
- The command to find SUID files on the system is `find /usr/bin -perm -4000 2>/dev/null` which could return files like `-rwsr-xr-x 1 root root 68208 Jan 1 /usr/bin/passwd`. The s in place of the usual x signifies that it is a SUID file.

**1. `sudo modprobe vboxdrv` and `sudo modprobe -r kvm_amd`**

`modprobe` loads or unloads kernel modules — think of kernel modules as plugins for the Linux kernel that add hardware support or features.

- `sudo modprobe vboxdrv` — loads the VirtualBox kernel module. This is what makes VirtualBox actually work — without this module loaded, VirtualBox can't create or run virtual machines. You ran this earlier when fixing your VirtualBox error.
- `sudo modprobe -r kvm_amd` — the `-r` flag means **remove**. `kvm_amd` is the AMD processor virtualization module. Removing it frees up virtualization resources — sometimes needed when VirtualBox and KVM conflict with each other.

---

**2. How to create a SUID file**

Two steps — create a file, then set SUID:

bash

```bash
# Create a script
echo '#!/bin/bash' > myscript.sh
echo 'whoami' >> myscript.sh

# Give it execute permission first
chmod 755 myscript.sh

# Now add SUID
chmod u+s myscript.sh

# Verify — you should see 's' in owner execute position
ls -la myscript.sh
# -rwsr-xr-x


`u+s` means: add (`+`) the SUID bit (`s`) to the user/owner (`u`) permissions.

You can also use numeric — SUID is represented by `4` added as a prefix:

bash

```bash
chmod 4755 myscript.sh   # same result

**3. Can a folder execute? Can it be SUID?**

Yes — but execute means something completely different for directories.

For **files**: execute means you can run it as a program.

For **directories**: execute means you can **enter** it with `cd`. Without execute permission on a directory, you cannot `cd` into it even if you can read it.

Try this:

bash

```bash
mkdir testfolder
chmod 644 testfolder    # remove execute
cd testfolder           # Permission denied
chmod 755 testfolder    # restore execute
cd testfolder           # works


**Can directories be SUID?** Technically yes, but it behaves differently and is rarely used. What you'll encounter more is **SGID on directories** — when set on a folder, new files created inside inherit the folder's group automatically. Useful for shared team directories.

---

**4. Permissions starting with `l`**

`l` means **symbolic link** — essentially a shortcut or alias pointing to another file or directory. Like a Windows shortcut.

```
lrwxrwxrwx 1 root root 4 Jan 1 /bin/sh -> dash
```

The `-> dash` tells you where the link points. The permissions on the link itself don't matter much — what matters is the permissions on the file it points to.

File types you'll see:

|Character|Type|
|---|---|
|`-`|Regular file|
|`d`|Directory|
|`l`|Symbolic link|
|`c`|Character device (keyboard, terminal)|
|`b`|Block device (hard drives)|

---

**5. SUID files highlighted in red**

Yes — that's `ls` with color output enabled. Red highlighting is a visual warning that the file has special permissions. It's a quick way to spot them when browsing directories.

However don't rely on color alone — always confirm with:

bash

```bash
ls -la | grep '^-..s'

Or the find command. Colors can be disabled or customized on different systems. In a real engagement you can't assume color output.
