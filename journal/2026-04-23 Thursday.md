- Using `lsof -i :631`, it zooms in to the services running on port 631. This is known as port specific investigation where you drill into what exactly you want.

- Assuming i'm an attacker that gained access as a system user, then i'd do the following

	2. Next I look at all the running processes that are root owned using the command `ps aux | grep 'root'` as they are potential targets for escalation
	3. Finally I run `ss -tulnp` to check for services that are listening on localhost but are not exposed externally as they are hidden attack surface that are only reachable from the inside
	The logic is:
	- First know **who you are**
	- Then know **what's running**
	- Then know **what's reachable**

### What I learned
- Permissions starting with l means a symbolic link, essentially a shortcut or alias pointing to another file or directory.
- Special permissions have symbols like ^-..s
- sort -k5 -n sorts column 5 numerically. `-k` specifies which **column** to sort by. `-n` means sort numerically instead of alphabetically. The columns are the texts seperated into fields by whitespaces.
- In the command `find /etc -name '*.conf' | xargs grep 'password'`, `xargs` takes the output of `find` and passes each filename as input to `grep`. This searches inside every `.conf` file in `/etc` for the word password. That's a real attacker command.
- Root owns system processes while user owns user processes hence if an attacker enters as a user, his main goal is to have access to the root-owned processes
```
Things attackers look for when runnning ps aux apart from root-owned processes
- Running services — web servers, databases, SSH daemons. Each one is a potential attack vector
- Credentials in process arguments — developers sometimes pass passwords as command line arguments like `python3 app.py --password=secret123`. These show up in `ps aux` in plain text
- Internal services — processes listening on ports not exposed externally, revealing hidden attack surface
```

- On `ss -tulnp` columns:

| Column               | Meaning                                                                                                                |
| -------------------- | ---------------------------------------------------------------------------------------------------------------------- |
| `Netid`              | Protocol — tcp or udp                                                                                                  |
| `State`              | `LISTEN` = waiting for connections, `UNCONN` = UDP open but not connected (UDP doesn't establish connections like TCP) |
| `Recv-Q`             | Data received but not yet processed — normally 0                                                                       |
| `Send-Q`             | Data queued to send — normally 0                                                                                       |
| `Local Address:Port` | What IP and port your machine is listening on — `0.0.0.0` means listening on all network interfaces                    |
| `Peer Address:Port`  | Who is connected to you — `*` means nobody yet                                                                         |
| `Process`            | Which program owns that port                                                                                           |
- An attacker runs `ss -tulnp` to answer three questions:
	1. What services are running? Each service is a potential vulnerability
	2. What's listening internally? Services bound to `127.0.0.1` aren't visible from outside but are reachable from inside — hidden attack surface
	3. What can I pivot to? A database on port 3306 listening only on localhost means it's reachable from inside — credentials found elsewhere could unlock it
- `ss -tulnp` shows you:
	- Which ports are listening
	- Which protocol (TCP/UDP)
	- Which process owns the port
- `lsof -i` shows you:
	- Everything `ss` shows
	- **Plus** the actual process name and PID
	- **Plus** active connections — not just listening ports but also established connections right now
	- **Plus** which user owns each connection
- The key difference is **active connections**. `ss -tulnp` shows what's waiting for connections. `lsof -i` shows what's actually connected right now.
- For an attacker this matters because:
	- `ss -tulnp` = what doors exist
	- `lsof -i` = who is currently walking through those doors

**Real scenario:** An attacker lands on a server. They run `lsof -i` and see an established connection from an admin's IP address. They now know:
	- An admin is actively logged in
	-  They should be careful — admin might notice suspicious activity
	- Or alternatively — they can try to hijack that session

Think of it as two levels of investigation:

Level 1 — `ss -tulnp` (the map) Shows you the landscape. What ports exist, what protocol, what state. Fast overview. You run this first to get the full picture of what's listening.

bash

```bash
ss -tulnp
# You see: tcp LISTEN 0.0.0.0:631  process: cupsd


Now you know port 631 exists and cupsd owns it.

Level 2 — `lsof -i :631` (the microscope) Zoom into a specific port you found interesting. Shows active connections, which user owns it, full process details.

bash

```bash
lsof -i :631
# You see: who is connected right now, from where


The connection between them:

`ss` answers: what exists? `lsof` answers: what's happening on it right now?

You use `ss` to build your list of interesting targets, then `lsof` to investigate each one deeply.

Real attacker workflow:

bash

```bash
# Step 1 - get the full map
ss -tulnp

# Step 2 - spot something interesting
# e.g. MySQL on 127.0.0.1:3306 — internal database

# Step 3 - investigate it
lsof -i :3306
# See: is anyone connected? which user? from where?

# Step 4 - cross reference with processes
ps aux | grep mysql
# See: what user runs MySQL? what are its arguments?
# Maybe credentials are visible in the arguments

Each tool feeds the next. That's the methodology.
